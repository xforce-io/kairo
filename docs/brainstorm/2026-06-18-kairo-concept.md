## 0. 起源:这不是新点子,是一次"觉察"的产品化

上一轮我们盯着 `前序项目` 看出一件事:**一整棵知识树(术语表、商业判断、可运行引擎),源头只是一个下午几段会议录音。** 真正有价值的不是录音,而是那条**"从脏、少、口语的源头,用纪律一层层逼出可靠知识"的路径**。

kairo 想做的,就是把这条路径从**一次性手工活**变成**可复用的引擎**:用户只要把 reference 丢进来,`step` 一下,知识自动往前长一格。

---

## 1. 核心心智模型:知识的「增量构建系统」

把它类比成大家都熟的东西,设计一下子就清楚了:

| kairo 概念 | 等价于 | 含义 |
|---|---|---|
| `step` 命令 | `make` / 调和循环(reconcile) | 扫当前状态,把每个能推进的节点**往前推一格** |
| reference(音频/docx/URL/xmind…) | 源文件(不可变、内容寻址) | 只读输入,丢进来就不改 |
| 处理流水线(ASR→转写→纪要→分析) | 构建规则(declared in/out, 幂等, 按 hash 缓存) | 每一格是一条可重跑的规则 |
| 多米诺骨牌 | 构建依赖 DAG 的边 | 上一格产物 ready → 触发下一格 |
| 宪法文件(constitution.md) | Makefile / 期望状态声明 | 声明"本 workspace 有哪些目标文件、来一条新记忆该怎么更新它们" |
| workspace 文档(understanding/assessment) | 派生产物(可重新生成) | 不是手写的真相,是从记忆**长出来**的 |
| 记忆/manifest 加一条 | 节点状态推进的记录 | "我有了一条关于这个 topic 的新记忆" |
| 分层定期归并 | **LSM-tree 分层合并 / B-tree 页索引** | 把日级纪要滚成周级、月级总结,工作记忆有界 |

**统一抽象:kairo = 一个跑在「依赖 DAG」上的调和循环。**
每个节点有状态(raw / 已转写 / 已纪要 / 已整合);`step` = 对所有"输入就绪且比产物新"的节点跑它的规则,推进一格。骨牌就是 DAG 的边。这天然解决了你提到的两件事:**一个 reference 被多个 topic 共用**(DAG 扇出)、**不同 topic 不同下游**(每个 topic 的宪法定义自己那片子图)。

这个模型直接继承了 前序项目 的命根子:**确定性、可重跑、每个产物都知道自己的源头(可追溯)。**

## 2. 关键构件(语音里提到的,逐个落位)

1. **Reference store(只读、内容寻址)** — 丢进来的原始载体。原始文件**不必跟 workspace 存一起**(它只是 reference,可在别处、可按 topic 专属地址)。按日期归档,像 lab notebook。
2. **Step 引擎(唯一驱动力)** — 一条命令,扫所有 topic,逐节点推进。所有动作都由 `step` 驱动,不存在"后台偷偷跑"。
3. **流水线模板(对一个数据单元的标准化处理)** — 原始素材 → ASR → 纪要/初步分析,有固定模板。这层产物是确定的、可 review 的。
4. **宪法文件(per-topic + 默认)** — 配置中枢。声明目标文件清单 + 更新协议。有默认宪法,每个 topic 可覆盖。
5. **记忆层(memory / manifest)** — 纪要产出后,往 topic 记忆加一条"新记忆",再触发 update workspace。
6. **分层归并(temporal compaction)** — 会议越攒越多 → bi-weekly/monthly 合并成高层总结。来新东西时**不必回看全部历史**,顺索引找相关旧文档即可。
7. **Per-topic 下游动作 / post-hook** — 不同 topic 总结后要更新的文件、要触发的 hook 不一样,可配置。

## 3. 必须现在拍板的设计决策(每个给选项 + 我的建议)

### D1. 资料 vs 纪要:一分两定吗?(你语音最后那一问,我认为是全场最关键)

两种输入**性质根本不同**:

| | 纪要流(event record) | 资料库(reference / knowledge base) |
|---|---|---|
| 时间性 | 有日期、追加式、按时间排 | 无内在时间锚,是"常驻知识" |
| 类比 | **日志 / 事件流** | **图书馆 / 查找表** |
| 归并 | 适合按时间分层归并(LSM) | 改了才重抽,按 topic/实体索引 |
| 角色 | 被整合进 understanding | 作为**上下文**喂给纪要处理 |

**建议:两条轴分开,但机制统一。**
都走同一条 `step` 流水线变成派生产物;区别只在**怎么归档、怎么归并**:纪要流 → 时间日志 → 分层归并;资料库 → topic 图书馆 → 改了才重抽,被多个纪要引用。

> **强证据:你的 `前序项目` 已经手工给出了这个答案。** 它早就把 `data/<会议>_<日期>/`(事件流)和 `references/`(常驻资料)**分开**了。所以"一分两定" → **是,而且你已经凭经验验证过了**。kairo 只是把这个你下意识做对的划分,变成显式机制。

### D2. LLM 在环 vs 确定性:海拔反转了,纪律怎么迁移?

⚠️ 注意一个反转:**rx2meal 的价值是"零 LLM、纯规则、可追溯";但 kairo 的流水线(ASR、写纪要、update understanding)天生是 LLM 活。** 它和 rx2meal 处在相反的海拔。

所以从 前序项目 继承的纪律**不是**"不许用 LLM",而是:
- 每个派生产物**挂源**(它从哪条 reference / 哪条记忆来)。
- **可重新生成 + 可 review + 可 diff**(像 前序项目 的 understanding/assessment 那样人能逐条看)。
- 借 前序项目 的 guardrail 思路:存 prompt+输入+输出、用 `manifest.counts` 让"漏抽/缩水"在 diff 里可见。

**这是 kairo 最该想清楚的硬骨头:怎么让 LLM 步骤保持可审计、不撒谎。**

### D3. `step` 的幂等 / 重跑语义

`step` 必须可反复跑。建议:**只在输入(内容 hash)变化时才推进**,产物 pin 住;LLM 的不确定性不能让 `understanding.md` 每跑一次就抖动一次。强制重算用显式 `re-step`。

### D4. workspace 文档的"合并冲突"

两条新记忆都要改 `understanding.md`、而你又手改过它 —— 这是真实的合并问题。两条路线:
- **(a) workspace 文档视为可从记忆重新生成** → 手改不该直接写文档,而是进宪法/记忆(推荐,和 D1"文档是派生物"一致)。
- **(b) 文档用稳定锚点的结构化格式,做结构化 merge**(更灵活但更复杂)。

> 顺带一个 前序项目 沉淀的洞见值得写进宪法:**事实 vs 判断分层**(understanding=事实、assessment=立场)。宪法应把"别让判断渗进事实文件"当一条一等规则。

## 4. 分期路线(别一口吃成大象)

这是个**多子系统**的东西(step 引擎 / 宪法 / 资料库 / 分层归并 / per-topic hook)。脑暴阶段先圈出**走通骨架的最小切片**:

- **MVP — 走路骨架(end-to-end 一个 topic)**:丢一个音频 reference → `step` 跑 ASR → 纪要 → 追加一条记忆 → 按宪法 update 一个 workspace 文件(understanding)。**没有**分层归并、**没有**多 topic。目标:证明"骨牌 + 宪法 + 可追溯"这条链真能自动走通——本质是把 前序项目 的核心循环用一个 topic 复现出来。
- **P2 — 多 topic**:per-topic 宪法、共享 reference(DAG 扇出)、**落地 D1 的资料库/事件流分轨**。
- **P3 — 分层归并**:时间维的 LSM 分层总结 / 页索引,让长历史有界。
- **P4 — 异构 loader + 富 hook**:URL / xmind / 飞书 / 更多 post-hook。

## 一句话给你自己

> kairo 的赌注是:**把"整理特定领域知识"这件高频但烦的事,降维成"丢个 reference + `step` 一下"。** 它继承 前序项目 的纪律(可追溯、可 review、派生物可重生),但海拔相反——它是**编排 LLM 的增量构建系统**,不是确定性规则引擎。先用一个 topic 走通骨架,再谈归并和多 topic。
